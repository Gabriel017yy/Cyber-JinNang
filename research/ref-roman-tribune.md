# 参考文档：罗马共和国保民官 (Tribune) 与绝对否决权 (Veto)

_这是为 chaoting-agent 架构设计中“审计/封驳”机制提供的历史理论支撑库。_

## 1. 什么是保民官 (Tribune of the Plebs)？
在古罗马共和国时期（公元前509年–前27年），国家的权力最初掌握在贵族（Patricians）手中，具体体现在元老院和两名执政官。为了保护平民（Plebeians）免受贵族的专横统治，罗马设立了**保民官 (Tribune of the Plebs)** 这一极其特殊的官职。

## 2. 绝对否决权 (Veto Power) 的本质
保民官的权力并不在于“制定政策”或“统领军队”，而在于**阻断与破坏**。

*   **Veto (我禁止)**：当元老院通过一项决议，或者执政官试图下达某个可能损害平民利益的命令时，保民官只需亲自到场，喊出一个词：**"Veto" (拉丁语意为“我禁止”/ I forbid)**。
*   **绝对阻断**：一旦保民官行使否决权，该项政令瞬间失效，无法执行，且不得上诉。
*   **无需建设性意见**：保民官不需要提供“更好的修改方案”，他们的职责就是发现风险并一剑封喉。这是一种纯粹的、单向的“防御与阻断机制”。

## 3. 为什么 Veto 机制对 AI 架构极具价值？
在设计复杂的多 Agent 协作系统时，我们通常习惯于让 Agent 互相讨论、修改对方的计划（类似于一种“共创”模式）。但历史经验表明，这种模式效率极低，容易陷入死循环（Oscillation）。

罗马人的智慧在于，将**“创造/规划者”**（元老院/执政官）和**“审计/风控者”**（保民官）在职能上彻底物理隔离：

1.  **极简的职责**：我们的 Auditor Agent 就像保民官，不需要知道怎么写代码或怎么做商业调研。它只拥有一个 Prompt：*“检查这份计划书，如果违反了用户的核心限制（预算超时、非法接口、逻辑断层），直接抛出 Veto 并附带理由，阻止计划流转。”*
2.  **强制返工**：Planner Agent 在收到 Veto 理由后，必须自行重新思考并生成新的计划，而不是和 Auditor Agent 进行无休止的“扯皮和互相修改”。
3.  **零冗余流转**：这使得代码控制流极其干净：`while (!approved) { plan = Planner.generate(); approved = Auditor.review(plan); }`。

这种基于 Veto 权力的单向制衡，是我们赛博决策引擎中 `Aegis (神盾核)` 机制的底层灵魂。
