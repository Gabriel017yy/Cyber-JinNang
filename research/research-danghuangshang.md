# 竞品研究：Danghuangshang (AI 朝廷) 架构分析

_最后更新：2026-04-28_

在正式开始编码前，我们对市面上现有的 "AI 朝廷" 方案 ([wanikua/danghuangshang](https://github.com/wanikua/danghuangshang)) 进行了充分的研究。本报告总结了其技术实现原理、目标场景，并与我们即将开发的 `chaoting-agent` 进行了深度对比，以明确我们的差异化和发挥空间。

---

## 1. Danghuangshang 项目概览

*   **项目定位**：基于 Discord/飞书 的多 Agent 协作 Bot 模板。
*   **目标场景**：团队协作、日常任务派发、开发辅助（如写代码、查账）。
*   **产出内容**：聊天群组中的 Bot 回复消息。
*   **核心理念**：“一行命令起王朝”，强调开箱即用、一键安装。用户在聊天软件中 @ 对应的“部门”来下达指令。

## 2. 核心技术实现：OpenClaw 框架

Danghuangshang **本身并不是一个从零开发的 Agent 框架**，而是高度依赖 [OpenClaw](https://github.com/openclaw/openclaw)（一个个人 AI 助手底层框架）的**配置模板**集合。

### 2.1 多 Agent 协作机制 (OpenClaw 底层原理)

其多 Agent 联动的核心不是代码层面的复杂调用图，而是通过 OpenClaw 提供的**消息路由**和**进程间通信 (IPC/Skill)** 实现的：

1.  **OpenClaw Gateway (网关)**:
    *   这是一个 Node.js 守护进程，负责连接 Discord、Slack、飞书等外部渠道 (Channels)。
    *   它维护着路由表 (Bindings)，根据消息的来源（哪个 Channel、哪个用户账号）将其分发给对应的 Agent。
2.  **独立的 Agent 进程**:
    *   兵部、户部、司礼监等，每个都是一个拥有独立会话 (Session)、独立记忆 (Memory) 和独立沙箱 (Docker) 的 Agent。
    *   每个 Agent 对应不同的模型配置（例如兵部用 Claude 3.5 Sonnet 写代码，礼部用 DeepSeek V3 写文案）。
3.  **部门联动 (Spawn & Send)**:
    *   Agent 之间的交流是通过 OpenClaw 提供的内置 Skill（工具）实现的。
    *   `sessions_spawn`：一个 Agent（如司礼监）可以调用此工具，创建一个子任务派发给另一个 Agent（如兵部）。
    *   `sessions_send`：Agent 之间发送消息通信。
    *   **触发机制**：最简单的方式是，用户或代理在聊天中直接 @另一个机器人账号（例如在 Discord 中，司礼监 Bot 的回复里 @了兵部 Bot，从而触发兵部 Bot）。

### 2.2 两种工作模式

1.  **司礼监调度模式（默认）**：
    *   用户 (皇帝) $\rightarrow$ @司礼监
    *   司礼监调用 "内阁" (其实是 Prompt 增强阶段) 生成执行计划。
    *   司礼监通过 `@提及` 派发任务给六部。
    *   六部执行完毕后，可能触发 "都察院" (QA) 审查。
2.  **直接指挥模式**：
    *   用户直接 `@兵部 写个登录 API`。绕过司礼监，直接单点调用。

## 3. 对比分析：为什么我们还有巨大空间？

仔细研究后发现，Danghuangshang 解决的问题和我们 `chaoting-agent` 解决的问题**完全不同**。这正是作为 AI PM 面试作品的**绝佳差异化亮点**。

| 维度 | Danghuangshang (竞品) | Chaoting-agent (我们的项目) |
| :--- | :--- | :--- |
| **本质形态** | 聊天机器人群组 (Discord Bot 集合) | **终端 CLI 决策引擎** |
| **底层技术** | 第三方框架 OpenClaw 的配置文件 | 基于 **pi-agent-core** 从零实现流水线代码 |
| **核心目的** | 执行具体杂项任务（写代码、查天气、发日报） | **解决“个人复杂决策辅助”问题**（如：要不要换工作？） |
| **交互方式** | 在群聊里 @ 不同的人聊天 | **“奏折式”结构化交互**，用户只做最终判断（准奏/驳回/追问） |
| **流程控制** | 依赖机器人在聊天群里互相 @，流程松散 | **代码级强控制**：太子分拣 $\rightarrow$ 中书省规划 $\rightarrow$ 门下省封驳循环 $\rightarrow$ 尚书省并行调度 |
| **制衡机制** | 主要是代码审查 (QA) | **核心机制**：门下省有权直接驳回中书省的计划，并给出理由重新生成 |
| **PM 价值展现** | 会配环境，会写 Prompt | **能设计复杂业务流，懂 Agent 编排，理解“可审计性”和“人机交互界限”** |

## 4. 研究结论

1.  **方向极其正确**：Danghuangshang 是一个“通用助手集合”，而我们需要的是一个“专业决策引擎”。两者虽然都借用了“朝廷”的隐喻，但内核完全不同。
2.  **无需照搬其架构**：我们不需要搞复杂的 Discord 机器人绑定和 Docker 沙箱。我们应该专注在 **Node.js/TypeScript 代码层面实现一套严密的、有审计路径的同步/异步执行流**。
3.  **下一步行动**：研究非常充分，证明了我们项目设计的独特性。现在我们可以毫无顾虑地回到 `DESIGN.md`，开始动手写 `package.json` 和初始化代码了。
