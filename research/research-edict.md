# 竞品研究：cft0808/edict 架构分析与对比

_最后更新：2026-04-28_

在研究了 `wanikua/danghuangshang` 之后，我们进一步调研了另一个同源且存在争议的项目：[`cft0808/edict`](https://github.com/cft0808/edict)。本报告将分析其架构特点、与 `danghuangshang` 的异同，并提炼对我们 `chaoting-agent` 项目的借鉴意义。

---

## 1. cft0808/edict 项目概览

*   **定位**：同样是基于 OpenClaw 框架的多 Agent 协作系统，也采用了“三省六部制”的隐喻。
*   **核心卖点**：强调“制度性审核 + 完全可观测 + 实时可干预”。
*   **架构扩展**：在基础的 OpenClaw 之上，开发了大量的 Python 后端服务（基于 FastAPI/SQLAlchemy/Redis）和 React 前端看板（“军机处看板”）。

## 2. 核心架构与功能 (与 danghuangshang 的区别)

虽然两个项目都叫“三省六部”，底层都用了 OpenClaw，但 `edict` 在业务流转和工程实现上走得更深：

### 2.1 引入了真正的“门下省”机制 (制度性审核)

这是 `edict` 最核心的设计差异：
*   **Danghuangshang**: 主要是“司礼监 $\rightarrow$ 内阁(Prompt优化) $\rightarrow$ 六部(执行)”。流程是**单向**的，缺乏强制的验证环节。
*   **Edict**: 实现了“中书(规划) $\rightarrow$ **门下(审议)** $\rightarrow$ 尚书(执行)”。门下省负责审查中书省的规划，并且**有权封驳（打回重做）**，形成了一个强制的质量把控闭环。

### 2.2 强大的工程化包装 (非纯 Prompt 配置)

*   **状态机校验**：`edict` 编写了 Python 脚本（如 `kanban_update.py`）来强制校验任务的状态流转，防止非法跳转（例如不能直接从“执行中”跳回“太子分拣”）。
*   **异步调度引擎**：引入了 Redis Streams 作为事件总线，使用了 DAG 编排器处理子任务依赖，以及具备指数退避重试机制的并行调度器。
*   **可视化看板 (Dashboard)**：提供了一个非常完整的 React 看板，实时监控任务状态（待办、执行中、审查中等）、Agent 心跳、Token 消耗等。

### 2.3 抄袭争议的可能来源

（*注：仅从技术实现角度推断，非最终结论*）
两个项目高度相似的底层都源于 OpenClaw 框架。`danghuangshang` 像是一个非常成熟的、开箱即用的 OpenClaw **配置集合与一键部署方案**；而 `edict` 更像是在 OpenClaw 基础上，为了实现严格的“三省流转”和看板监控，而**外挂了一套复杂的 Python 后端系统**。

如果存在抄袭争议，大概率是因为两者在角色设定（太子、三省、六部对应具体的 Agent 职责）和部分 Prompt 模板（`SOUL.md`）上出现了高度重合。但从工程架构上看，`edict` 开发了大量额外的后端控制逻辑和前端 UI。

---

## 3. 对 Chaoting-agent 的借鉴意义

研究 `edict` 之后，我们对 `chaoting-agent` 的设计方向更加确信了：

1.  **“门下省”的封驳机制是灵魂**：
    `edict` 证明了“强制审核与打回重做”在复杂 Agent 编排中的重要性。我们的 `chaoting-agent` 也将门下省作为核心机制，这不仅是历史隐喻，更是提升输出质量的工程手段。

2.  **我们不需要沉重的 Web 看板和异步队列**：
    *   `edict` 为了做成一个“企业级/团队级”的任务管理系统，引入了 Redis、SQLAlchemy、React 等全栈技术。
    *   **我们的优势在于“轻巧”和“CLI 交互”**。我们定位是**个人的复杂决策辅助**工具。我们将通过纯 TypeScript 和 `pi-agent-core` 的进程内调用（如 `Promise.all()`），在终端中实现高效的同步/异步流转，并通过“奏折”形式一次性呈现结果。

3.  **强化“决策”属性**：
    `edict` 的例子（如设计 API、写代码审查）依然偏向“执行力”。我们的 `chaoting-agent` 必须死死咬住“**复杂决策**”这个场景（例如：评估两个 Offer 的优劣、分析是否应该转行），通过六部从不同维度（财务、法律、风险、执行力等）给出建议，最终汇总成决策奏折。

## 4. 总结

`edict` 和 `danghuangshang` 都是用“三省六部”包装的多 Agent 执行系统，前者工程化更重，实现了审核机制。我们的 `chaoting-agent` 则是一个**纯 CLI、轻量级、专注于“个人决策分析”**的工具，我们将在代码层面用极简优雅的方式实现“封驳”和“并行”，在面试中展现出克制且精准的架构设计能力。
